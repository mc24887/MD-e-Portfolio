Module Overview

This study delivered a practical assessment pipeline across host and web surfaces using Nmap, OpenVAS, OWASP ZAP and Wireshark. Automated results were validated at packet level and converted into a clear, risk‑prioritised remediation plan aligned to best practice.

Key Submissions & Evidence

1) Service & TLS enumeration (Nmap)

Enumerated open services, versions and TLS posture to expose weak ciphers and legacy endpoints.

nmap -sV -p 80,443,8080 --script ssl-enum-ciphers --reason target.example.com
nmap -sC -sV -O -Pn target.example.com
# Targeted UDP high‑value checks (SNMP/DNS where applicable)
nmap -sU -p 53,123,161 --script=banner target.example.com
  • Findings: HTTP on 8080 (plaintext), TLS on 443 offering legacy ciphers; server version leakage.
  • Controls: force TLS‑only, HSTS, disable weak ciphers, minimise banner leakage.

2) Vulnerability scanning & triage (OpenVAS)

Executed a full scan, then triaged by exploitability and business impact (not CVSS alone).

IssueEvidenceRiskRemediation
Weak TLS cipher suites OpenVAS + Nmap ssl-enum-ciphers High (MITM, downgrade) Disable RC4/3DES, prefer TLS 1.2/1.3; set modern server profile
Exposed server banner HTTP responses leak exact product/version Medium (recon → targeted exploits) Suppress/obfuscate server banner; centralise through a hardened reverse proxy
HTTP on 8080 Clear‑text credentials observed in capture High (credential exposure) Redirect 80/8080 → 443; set HSTS with preload and includeSubDomains

3) Web application assessment (OWASP ZAP)

Baseline scan for missing headers and input handling issues. Hardened responses and re‑tested.

# ZAP baseline example
zap-baseline.py -t https://target.example.com -r zap-baseline.html -m 3
# Manual exploration: proxy browser through ZAP, then active scan selected contexts.

Header hardening:

X-Frame-Options: DENY
Content-Security-Policy: default-src 'self'
Referrer-Policy: no-referrer
X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

4) Packet‑level verification (Wireshark)

Confirmed issues and fixes on the wire to eliminate false positives.

# Filters used
tcp.port == 80 and http
http.request.method == "POST"
tls.handshake.version
frame contains "password="
  • Before: plaintext POST credentials visible on 8080.
  • After: TLS‑only, HSTS enforced, no sensitive content observable in clear‑text.

Reflection

Combining automated tooling with packet‑level verification increased the signal‑to‑noise ratio, reduced rework and produced remediation actions that operations could apply quickly. Writing the plan in business language (impact & effort) improved stakeholder buy‑in and shortened turnaround.

  • Vulnerability scanning, triage and risk‑centred prioritisation.
  • HTTP header hardening and CSP tuning.
  • Packet capture, filtering and evidence‑based validation.
